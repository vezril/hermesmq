## Context

HermesMQ is event-sourced/CQRS on Pekko. Feature 4 will model Topics and Subscriptions as `EventSourcedBehavior` actors, whose command handler validates a command against in-memory state and emits events, and whose event handler folds events into state. Feature 3 builds exactly that write-side core **as pure functions, with no Pekko Persistence**, so the domain rules are exhaustively unit-testable and Feature 4 becomes a thin actor wrapper. FP and clean code are explicit project values.

## Goals / Non-Goals

**Goals:**
- Immutable, validated value types: `TopicId`/`SubscriptionId`/`MessageId`/`AckId`, `Message`, `AckDeadline`.
- Sealed command and event ADTs for the Topic and Subscription aggregates.
- Pure `decide(state, command): Either[Rejection, List[Event]]` and `evolve(state, event): State` for each aggregate, fully test-driven.
- An explicit `Rejection` type for every invalid command.

**Non-Goals:**
- Any Pekko Persistence / `EventSourcedBehavior` / snapshots (Feature 4).
- Delivery scheduling, redelivery, ack-deadline expiry timers, fan-out from topic to subscriptions (later features). `RecordDelivery` here only marks a message outstanding; *who* calls it and *when* is out of scope.
- Projections / read models / gRPC / serialization wire formats (later).

## Decisions

**Pure decide/evolve split (the "decider" pattern).**
Each aggregate is an object with `decide(state, cmd): Either[Rejection, List[Event]]` and `evolve(state, evt): State`. This is precisely the shape Pekko's `EventSourcedBehavior(commandHandler, eventHandler)` needs, so Feature 4 wraps it with near-zero glue. Both functions are **total** — never throw; all failure is a `Left(Rejection)` or a well-defined state transition. *Alternative:* put logic directly in actors now — rejected: couples domain rules to Pekko and makes them far harder to test.

**Determinism: commands carry generated values; decide never reads a clock or RNG.**
`decide` must be referentially transparent (so event derivation is reproducible and testable). Anything non-deterministic — `MessageId`, publish timestamp, `AckId`, the concrete deadline instant — is generated by the caller (the actor, in Feature 4) and passed *in* via the command / `Message`. e.g. `Publish(message)` carries a fully-formed `Message`; `RecordDelivery(ackId, messageId, deadline)` carries the ids. *Alternative:* inject a `Clock`/`IdGen` into `decide` — heavier; carrying values in commands is the simplest total, pure design and mirrors how the actor will supply them.

**Identifiers as opaque types with smart constructors.**
Scala 3 `opaque type TopicId = String` with a validating `TopicId.from(String): Either[ValidationError, TopicId]` gives zero-overhead, non-interchangeable, non-empty-guaranteed ids. Distinct id types can't be swapped by accident. *Alternative:* case-class wrappers — fine but add allocation; plain `String` — unsafe and interchangeable.

**Validation via `Either[ValidationError, A]` smart constructors.**
Value types expose `from(...)` returning `Either`; no exceptions for invalid input. Domain rejections use a separate `Rejection` ADT. Keeping *construction* errors (`ValidationError`) distinct from *command* rejections (`Rejection`) keeps the two failure vocabularies clear. *Alternative:* a validation library (cats `Validated`) — not worth a dependency for this scope; standard `Either` suffices.

**Message envelope: immutable, non-empty payload, defensively-copied attributes.**
`Message(id, payload: IndexedSeq[Byte] or Array copied, attributes: Map[String,String], publishTime: Instant)`. Attributes stored as an immutable `Map`; payload held immutably (copy on construction if backed by a mutable array). Empty payload rejected. *Alternative:* allow empty payloads — rejected for now (a broker message should carry a body); revisit if a use case appears.

**Subscription state carries an outstanding map.**
`SubscriptionState(exists, topicId, outstanding: Map[AckId, Outstanding(messageId, deadline)])`. `RecordDelivery` adds; `Acknowledge` removes; `ModifyAckDeadline` updates. This is the minimal state needed to make ack/modify meaningful without implementing delivery. *Alternative:* omit outstanding tracking now — rejected: then ack/modify have nothing to act on and can't be tested meaningfully.

## Risks / Trade-offs

- **Modeling ack without delivery feels partial** → mitigated by `RecordDelivery` as the single, minimal seam that marks a message outstanding; real delivery/redelivery is layered on later without changing these events.
- **decide/evolve contract may shift when Feature 4 wraps it** → chosen signatures deliberately match Pekko's `CommandHandler`/`EventHandler`; if adjustment is needed it's a rename, not a redesign.
- **Opaque types + Scala 3 given/derivation ergonomics** → keep smart constructors and equality simple (opaque over `String` compares as `String`); avoid premature typeclass machinery.
- **Payload representation (Array vs IndexedSeq)** → use an immutable representation to guarantee the "attributes/payload can't mutate after construction" scenarios; accept a copy cost at construction.
- **Over-scoping "basic"** → three small capabilities, each a handful of pure functions; if it feels large the Subscription slice (record/ack/modify) is the natural place to trim.

## Migration Plan

Pure, additive code with no runtime wiring; TDD order per capability:
1. `domain-model`: value types + smart constructors (ids, `Message`, `AckDeadline`) — tests first.
2. Command/event ADTs + `Rejection` (compile-time shape; exercised by aggregate tests).
3. `topic-lifecycle`: `Topic.decide`/`evolve` — create, publish, rejections.
4. `subscription-lifecycle`: `Subscription.decide`/`evolve` — create, record-delivery, acknowledge, modify-deadline, rejections.
5. Full suite green; no README runtime changes needed (optionally note the domain package).

Rollback: additive files only; reverting removes the `domain` package with no effect on the running service.

## Open Questions

- Payload type: `Vector[Byte]` vs a thin immutable `ByteString`-like wrapper. Assuming an immutable byte sequence for now; Feature 4 serialization may prefer Pekko `ByteString`.
- Should `Rejection` be a single shared ADT or per-aggregate? Leaning shared with clearly-named cases (`TopicAlreadyExists`, `TopicNotFound`, `UnknownAckId`, …).
- Whether `CreateSubscription` should validate that the referenced `TopicId` exists — cross-aggregate check that belongs to the actor/coordination layer (Feature 4), not the pure Subscription aggregate, so it is intentionally *not* enforced here.
